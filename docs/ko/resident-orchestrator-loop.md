# 상주 오케스트레이터와 자율 Loop: 멀티 레포 운영 층

> 한국어 문서. English: [../en/resident-orchestrator-loop.md](../en/resident-orchestrator-loop.md)

이 하네스의 `.agent` 층이 "한 레포 안에서 에이전트가 어떻게 일하나"를 다룬다면,
이 문서는 그 위층을 다룬다: **레포 여러 개를 혼자 관리하는 사람이, 공통 부모
폴더에 마스터 에이전트를 상시 띄워 두고 실행을 워커에게 재하청하는 구조**와,
그 마스터가 도는 자율 Loop의 설계다. 실제 서비스(백엔드 모노레포 + 프론트 레포,
라이브 운영)에서 하루 종일 굴리며 사고까지 겪은 기록을 일반화했고, 근거가 되는
외부 리서치를 말미에 인용한다.

도구 이름은 예시일 뿐이다. 상주 터미널은 tmux여도 되고, 메모리는 파일이어도
되고, 워커는 어느 CLI 에이전트여도 된다. 채워야 하는 것은 역할이다.

## 1. 다섯 역할

| 역할 | 요건 | 없으면 생기는 일 |
|---|---|---|
| 상주 위치 | 레포들의 공통 부모 폴더에서 실행 | 교차 영향(API 계약 변경→프론트 재생성)을 놓친다 |
| 영구 기억 | 세션 시작 시 규칙·결정·함정 자동 주입 | 새 세션마다 같은 실수를 반복한다 |
| 감시 루프 | ref/dirty/워크트리/MR 변화를 주기 diff | 사람이 자리 비운 사이 변화를 아무도 모른다 |
| 하청 + 게이트 | 구현은 워커에, 산출물은 실측 재검증, 파괴 행위는 사람 게이트 | 오류가 증폭되거나 워커가 폭주한다 |
| 승계 | 세션 종료 전 얇은 핸드오프 + 킥오프 프롬프트로 후임 스폰 | 세션이 죽을 때마다 운영 지식이 리셋된다 |

## 2. 기억 설계: 3층 분리와 SSOT 이원화

컨텍스트는 유한하고, 길어질수록 회상 정확도가 떨어진다(context rot, [R1]).
그래서 기억은 컨텍스트 밖 저장소로 빼되, 층을 나눈다.

| 층 | 담는 것 | 저장소 | 주입 시점 |
|---|---|---|---|
| 규칙·결정·함정 | "무조건 이렇게" / "이거 하지 마" | 메모리 저장소(예: beads `bd remember`) | 매 세션 자동 |
| 작업 단위 서사 | 이 이슈가 왜 이 상태인지 | 이슈 노트 | 해당 이슈 조회 시 |
| 세션 서사 | 지금 열린 트랙 | 핸드오프 문서 1장 | 승계 시 1회 |

장기 작업 에이전트의 표준 패턴도 같다: 컨텍스트 밖에 구조화된 노트를 계속
적립하고 [R3], 세션 경계에서는 요약(compaction)으로 승계한다 [R2]. 실제 OSS
코퍼스 조사에서도 파일 기반 지속화가 경험적 표준이다(70개 프로젝트 중 순수
프롬프트-윈도우 방식은 4.3%뿐) [R5].

여기에 하나를 더 지켜야 구조가 오래 간다. **SSOT 이원화**: 의도·계획의 원본은
인간 소유 문서에, 실행 상태·사실의 원본은 메모리 저장소에 두고, 서로를 복제하지
않는다(포인터만). 같은 내용이 두 곳에 사는 순간부터 드리프트가 시작된다 —
선행 하네스의 `.agent` 파일들이 코드 실상과 어긋난 stale 사례가 이것이다.

## 3. 자율 Loop: 듀티 사이클 6단계

마스터가 깨어날 때마다(주기 wakeup 또는 이벤트) 도는 상태 기계.

1. **SENSE** — 감시 실측. 모니터 생존은 프로세스 실측으로 확인한다(런타임의
   태스크 조회는 세션 경계에서 거짓 사망 신호를 낼 수 있다 — 실측 2회 연속
   부재일 때만 재장착, 중복 장착 금지). 레포 기준선 diff, 워커 상태 확인.
   워커의 침묵은 사망이 아니다 — 터미널 활동을 보고 판단한다.
2. **TRIAGE** — 이벤트 분류. 자기 행위의 메아리는 무시, 예상된 변화는 기준선
   갱신, 설명 안 되는 변화는 즉시 사람에게 보고(자동 대응 금지).
3. **WORK** — 유휴 시 작업 선별. 이슈 트래커 ready 큐에서 우선순위순으로,
   게이트가 필요 없는 작업만 자발 착수. 착수 전 의무: 관련 결정 메모리 검색
   (닫힌 논점 재개봉 금지). 위임 시 작업 명세에 목표·산출 형식·도구 범위·
   경계를 전부 명시한다 — 모호한 위임은 중복·오해석 작업을 만든다 [R9].
   분해는 문제 유형이 아니라 **컨텍스트 경계** 기준으로 한다(플래너/테스터
   같은 역할 분할은 컨텍스트 공유 비용만 키운다) [R12].
4. **VERIFY** — 산출물 실측 재검증. 워커의 완료 주장을 믿지 않고 diff·테스트·
   실행으로 재확인한다. 검증 게이트에는 명시적 기준을 줘야 한다 — "괜찮아?"
   수준의 기준 없는 검증자는 고무도장이 된다 [R10]. 리뷰 에이전트의 지적도
   코드로 재확인한다(오검 기각 사례 실존).
5. **RECORD** — 이슈 노트, 티켓, 함정 메모리 적립(사고·오판마다 1건).
6. **PACE** — 다음 주기 self-pacing. 활성 트랙 있으면 짧게(10~15분), 유휴면
   길게(25~30분), 사람이 대화 중이면 루프는 배경으로.

### 하드 스톱 (안전 정지 조건)

종료 조건은 일급 설계 요소다 — 시간 예산과 명시적 종료 조건 없이 반응형
루프를 돌리면 폭주한다 [R11].

- 같은 작업 연속 실패 3회 → 중단, 사람에게 에스컬레이션(circuit breaker).
- 게이트 항목(머지·push·prod·삭제·설정 변경)은 어떤 경로로도 자발 실행 금지.
- 컨텍스트 사용량 임계(≈70%) → 신규 착수 중단, 핸드오프 작성 모드로 전환.
- 이상 행적 발견 → WORK 진입 금지, 보고 후 대기.
- 워커 폭주 징후(스코프 밖 파일, 무단 게시, 유령 프로세스) → kill + 산출물
  격리 + 보고. 실측 근거: "읽기 전용" 지시만 받은 리뷰 워커가 쓰기 도구를
  가진 채 무단 게시·프로세스 스폰까지 간 사고가 실제로 있었다. 지시는
  강제력이 없다 — 도구 제한으로 강제한다. OSS 조사에서도 컨테이너 격리가
  있는 프로젝트는 100%가 정책 엔진을 함께 갖춘다(격리와 거버넌스는 같이
  진화한다) [R7].

### 규모·비용 규칙

멀티에이전트는 기본값이 아니다. 등가 작업에서 단일 에이전트 대비 3~10배
토큰을 쓰므로 [R12], 병렬성이 실제로 이득인 고가치 작업에만 fan-out한다.
오케스트레이터에는 명시적 effort-scaling 규칙을 심는다(단순 조회 1에이전트,
대규모 조사에만 10+) — 이 규칙이 없던 초기 시스템은 단순 질의에 50개
서브에이전트를 띄웠다 [R8]. 서브에이전트는 깨끗한 컨텍스트로 돌리고 응축
요약(1~2천 토큰)만 회수한다 [R4]. 그리고 모델 비용 라우팅: 판단·검수는
주 모델, 대량 실행·리서치는 별도 과금 계열의 워커로. (이 문서를 만들던 날,
리서치 fan-out을 주 모델로 돌려 계정 크레딧을 태운 실사고가 규칙의 출처다.)

## 4. 위임 프로토콜 체크리스트

- 워커 스폰 후 **수신 검증까지가 디스패치다**: TUI idle 확인 → 주입 →
  터미널 tail에 태스크 텍스트가 실제로 보이는지 실측. 주입은 셸 초기화·
  자동 업데이트와 레이스해서 조용히 유실될 수 있다(3회 실측).
- 작업 명세에 반드시: 목표, 대상 경로(워크트리), 산출 형식, 금지사항
  (push·외부 API·스코프 밖 파일), 완료 보고 형식.
- 격리: 병렬 파일 변경은 워커별 워크트리. 베이스 브랜치는 명시(기본 브랜치
  포인터가 릴리스 브랜치를 가리키는 레포가 실존한다).
- 장수 워커 vs 일회성 워커: 맥락 축적이 이득인 반복 도메인 작업은 장수
  워커에, 단발 작업은 깨끗한 컨텍스트의 일회성 워커에 [R13].

## 5. 근거 (리서치 인용)

방법 주기: 아래는 5각도 병렬 검색 → 23개 소스 → 115개 주장 추출 → 표본
검증으로 수집했다. [R1~R4]는 3표 적대 검증을 통과했고, 나머지는 1차 소스
직접 인용이지만 투표 검증이 중단돼(비용 사고) 미완이다 — 소스 신뢰도로만
채택했다.

- [R1] context rot — 컨텍스트가 길수록 회상 정확도 하락. Anthropic, "Effective
  context engineering for AI agents" (3-0 검증)
- [R2] 세션 경계 승계 = compaction(요약 후 새 컨텍스트 재시작). 같은 글 (2-1)
- [R3] 장기 작업은 컨텍스트 밖 구조화 노트 적립. 같은 글 (3-0)
- [R4] 서브에이전트는 깨끗한 컨텍스트 + 1~2천 토큰 응축 요약 회수. 같은 글 (3-0)
- [R5] 70개 OSS 프로젝트 코퍼스: 파일 지속화가 경험적 표준(순수 프롬프트-윈도우
  4.3%). arXiv:2604.18071
- [R6] 위임 복잡도와 컨텍스트 관리 정교함의 강한 동시 발생(오케스트레이터-워커
  프로젝트의 85%가 파일 지속화). arXiv:2604.18071
- [R7] 실행 격리와 안전 거버넌스는 동시 진화(컨테이너 격리 프로젝트의 100%가
  정책 엔진 보유, lift 3.4). arXiv:2604.18071
- [R8] 폭주 실사례: effort-scaling 규칙 부재 시 단순 질의에 서브에이전트 50개.
  Anthropic, "Multi-agent research system"
- [R9] 모호한 위임 → 중복·오해석. 명세에 목표·형식·도구·경계 필수. 같은 글
- [R10] 기준 없는 검증자는 고무도장 — 게이트에 명시적 기준 필요. Anthropic
  블로그, "Multi-agent coordination patterns"
- [R11] 반응형 루프 폭주 방지 = 일급 종료 조건(시간 예산 + 명시 조건). 같은 글
- [R12] 멀티에이전트 = 단일 대비 3~10배 토큰, 분해는 컨텍스트 경계 기준.
  Anthropic 블로그, "Building multi-agent systems: when and how"
- [R13] 장수 워커의 도메인 특화 축적 vs 일회성 서브에이전트. Anthropic 블로그,
  "Multi-agent coordination patterns"
- 보조: 13개 OSS 코딩 에이전트 스캐폴드 분석 — 제어 구조는 5개 루프 프리미티브의
  합성이며 11/13이 다층 합성. arXiv:2604.03515

## 6. 하루 운영에서 나온 함정 목록 (실측)

- 런타임 태스크 조회의 거짓 사망 신호 → 살아 있는 모니터를 죽인 사고. 생존
  확인은 프로세스 실측으로.
- 프롬프트 주입과 셸 초기화의 레이스 → 태스크 유실 3회. 수신 검증 의무화.
- 배포 설정 리뷰를 diff 범위로 한정 → 이미지 생산 경로(compose) 누락을 리뷰
  2기 + 재검증으로도 못 잡음. 리뷰 스코프는 "기능의 동작 전제 전체".
- 리서치 fan-out을 주 모델로 실행 → 크레딧 소진. 비용 라우팅 규칙 명문화.
- 결정 메모리 미참조 제안 → 닫힌 논점 재개봉. "제안 문장을 쓰기 직전이 검색
  타이밍."
