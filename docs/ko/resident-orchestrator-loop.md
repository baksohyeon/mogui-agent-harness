# 상주 오케스트레이터와 자율 Loop: 멀티 레포 운영 층

> 한국어 문서. English: [../en/resident-orchestrator-loop.md](../en/resident-orchestrator-loop.md)

이 하네스의 `.agent` 층이 "한 레포 안에서 에이전트가 어떻게 일하나"를 다룬다면,
이 문서는 그 위층을 다룬다: **레포 여러 개를 혼자 관리하는 사람이, 공통 부모
폴더에 마스터 에이전트를 상시 띄워 두고 실행을 워커에게 재하청하는 구조**와,
그 마스터가 도는 자율 Loop의 설계다. 실제 서비스(백엔드 모노레포 + 프론트 레포,
라이브 운영)에서 하루 종일 굴리며 사고까지 겪은 기록을 일반화했고, 근거가 되는
외부 리서치를 말미에 인용한다. 출처 원문과 검증일은 [에이전틱 하네스
엔지니어링 리서치 데이터](research/agentic-harness-engineering.md)에 모아 두었고,
현지 운영에서 직접 관찰한 내용은 `[OBS]`로 표시한다.

도구 이름은 예시일 뿐이다. 상주 터미널은 tmux여도 되고, 메모리는 파일이어도
되고, 워커는 어느 CLI 에이전트여도 된다. 채워야 하는 것은 역할이다.
이식 가능한 디스패치는 워커 어댑터가 동일한 수신 계약을 제공해야 한다:
준비·수락 상태, 태스크 식별자와 대상, 수신 확인(ack), timeout, 실패 결과다.
tmux/TUI idle과 terminal tail은 그중 한 어댑터일 뿐이며, 파일 기반이나 다른
CLI 워커는 수신 확인 파일·이벤트·상태 조회로 동등한 정보를 제공할 수 있다.

## 1. 다섯 역할

| 역할 | 요건 | 없으면 생기는 일 |
|---|---|---|
| 상주 위치 | 레포들의 공통 부모 폴더에서 실행 `[AHE-A3] [OBS]` | 교차 영향(API 계약 변경→프론트 재생성)을 놓친다 `[OBS]` |
| 영구 기억 | 세션 시작 시 규칙·결정·함정 자동 주입 `[AHE-A2]` | 새 세션마다 같은 실수를 반복한다 `[OBS]` |
| 감시 루프 | ref/dirty/워크트리/MR 변화를 주기 diff `[AHE-C1] [AHE-C5]` | 사람이 자리 비운 사이 변화를 아무도 모른다 `[OBS]` |
| 하청 + 게이트 | 구현은 워커에, 산출물은 실측 재검증, 파괴 행위는 사람 게이트 `[AHE-B1] [AHE-C3]` | 오류가 증폭되거나 워커가 폭주한다 `[OBS]` |
| 승계 | 세션 종료 전 얇은 핸드오프 + 킥오프 프롬프트로 후임 스폰 `[AHE-A1] [AHE-A4]` | 세션이 죽을 때마다 운영 지식이 리셋된다 `[OBS]` |

## 2. 기억 설계: 3층 분리와 SSOT 이원화

컨텍스트는 유한하고, 길어질수록 회상 정확도가 떨어진다(context rot, `[AHE-A1]`).
그래서 기억은 컨텍스트 밖 저장소로 빼되, 층을 나눈다.

| 층 | 담는 것 | 저장소 | 주입 시점 |
|---|---|---|---|
| 규칙·결정·함정 | "무조건 이렇게" / "이거 하지 마" | 메모리 저장소(예: beads `bd remember`) `[AHE-B5]` | 매 세션 자동 |
| 작업 단위 서사 | 이 이슈가 왜 이 상태인지 | 이슈 노트 | 해당 이슈 조회 시 |
| 세션 서사 | 지금 열린 트랙 | 핸드오프 문서 1장 | 승계 시 1회 |

장기 작업 에이전트에 공개된 패턴도 같다: 컨텍스트 밖에 구조화된 노트를 계속
적립하고 `[AHE-A1] [AHE-A2]`, 세션 경계에서는 요약(compaction)으로 승계한다
`[AHE-A1]`. OpenHands, LangGraph, Claude 메모리 문서는 각각 이벤트·체크포인트·
파일 기반 지속화의 구체적인 구현을 보여 준다 `[AHE-A2] [AHE-A4] [AHE-A5] [AHE-C2]`.

여기에 하나를 더 지켜야 구조가 오래 간다. **SSOT 이원화**: 의도·계획의 원본은
인간 소유 문서에, 실행 상태·사실의 원본은 메모리 저장소에 두고, 서로를 복제하지
않는다(포인터만). 같은 내용이 두 곳에 사는 순간부터 드리프트가 시작된다 —
선행 하네스의 `.agent` 파일들이 코드 실상과 어긋난 stale 사례가 이것이다
`[OBS]`. 드리프트를 줄이기 위한 짧은 라우터·구조화 문서·기계적 최신성 검사는
OpenAI가 공개한 하네스 경험과도 맞닿아 있다 `[AHE-A3]`.

## 3. 자율 Loop: 듀티 사이클 6단계

마스터가 깨어날 때마다(주기 wakeup 또는 이벤트) 도는 상태 기계.

1. **SENSE** — 감시 실측. 모니터 생존은 프로세스 실측으로 확인한다(런타임의
   태스크 조회는 세션 경계에서 거짓 사망 신호를 낼 수 있다 — 실측 2회 연속
   부재일 때만 재장착, 중복 장착 금지). 레포 기준선 diff, 워커 상태 확인.
   워커의 침묵은 사망이 아니다 — 터미널 활동을 보고 판단한다 `[AHE-C1] [AHE-C5] [OBS]`.
2. **TRIAGE** — 이벤트 분류. 자기 행위의 메아리는 무시, 예상된 변화는 기준선
   갱신, 설명 안 되는 변화는 즉시 사람에게 보고(자동 대응 금지).
3. **WORK** — 유휴 시 작업 선별. 이슈 트래커 ready 큐에서 우선순위순으로,
   게이트가 필요 없는 작업만 자발 착수. 착수 전 의무: 관련 결정 메모리 검색
   (닫힌 논점 재개봉 금지). 위임 시 작업 명세에 목표·산출 형식·도구 범위·
   경계를 전부 명시한다 — 모호한 위임은 중복·오해석 작업을 만든다 `[AHE-B2] [OBS]`.
   분해는 문제 유형이 아니라 **컨텍스트 경계** 기준으로 한다(플래너/테스터
   같은 역할 분할은 컨텍스트 공유 비용만 키운다) `[AHE-B2] [AHE-B3]`.
4. **VERIFY** — 산출물 실측 재검증. 워커의 완료 주장을 믿지 않고 diff·테스트·
   실행으로 재확인한다. 검증 게이트에는 명시적 기준을 줘야 한다 — "괜찮아?"
   수준의 기준 없는 검증자는 고무도장이 된다 `[AHE-B1]`. 리뷰 에이전트의 지적도
   코드로 재확인한다(오검 기각 사례 실존) `[OBS]`.
5. **RECORD** — 이슈 노트, 티켓, 함정 메모리 적립(사고·오판마다 1건).
6. **PACE** — 다음 주기 self-pacing. 활성 트랙 있으면 짧게(10~15분), 유휴면
   길게(25~30분), 사람이 대화 중이면 루프는 배경으로 `[OBS]`.

### 하드 스톱 (안전 정지 조건)

종료 조건은 일급 설계 요소다 — 시간 예산과 명시적 종료 조건 없이 반응형
루프를 돌리면 폭주한다 `[AHE-B1] [AHE-C3] [AHE-C4]`.

- 같은 작업 연속 실패 3회 → 중단, 사람에게 에스컬레이션(circuit breaker) `[OBS]`.
- 게이트 항목(머지·push·prod·삭제·설정 변경)은 어떤 경로로도 자발 실행 금지 `[AHE-C1] [AHE-C3] [OBS]`.
- 컨텍스트 사용량 임계(≈70%) → 신규 착수 중단, 핸드오프 작성 모드로 전환 `[OBS]`.
- 이상 행적 발견 → WORK 진입 금지, 보고 후 대기.
- 워커 폭주 징후(스코프 밖 파일, 무단 게시, 유령 프로세스) → kill + 산출물
  격리 + 보고. 실측 근거: "읽기 전용" 지시만 받은 리뷰 워커가 쓰기 도구를
  가진 채 무단 게시·프로세스 스폰까지 간 사고가 실제로 있었다. 지시는
  강제력이 없다 — 도구 제한으로 강제한다. 공개 구현은 격리와 함께 정책·보안
  제어를 둔다 `[AHE-B3] [AHE-A5] [AHE-C1]`. 이 사고와 대응은 운영 실측이다 `[OBS]`.

### 규모·비용 규칙

멀티에이전트는 기본값이 아니다. Anthropic은 자사 멀티에이전트 리서치 시스템이
일반 대화 대비 약 15배의 토큰을 썼다고 보고하므로 `[AHE-B2]`, 병렬성이 실제로
이득인 고가치 작업에만 fan-out한다.
오케스트레이터에는 명시적 effort-scaling 규칙을 심는다(단순 조회 1에이전트,
대규모 조사에만 10+) `[OBS]` — 이 규칙이 없던 초기 시스템은 단순 질의에 50개
서브에이전트를 띄웠다 `[AHE-B2]`. 서브에이전트는 깨끗한 컨텍스트로 돌리고
응축 요약만 회수한다 `[AHE-B2] [AHE-B3]`. 그리고 모델 비용 라우팅: 판단·검수는
주 모델, 대량 실행·리서치는 별도 과금 계열의 워커로. (이 문서를 만들던 날,
리서치 fan-out을 주 모델로 돌려 계정 크레딧을 태운 실사고가 규칙의 출처다.)

## 4. 위임 프로토콜 체크리스트

- 워커 스폰 후 **수신 검증까지가 디스패치다**: 워커 어댑터에서 준비·수락
  상태를 확인하고 태스크 식별자·대상을 기록한 뒤, 같은 정보가 담긴 수신
  확인(ack)을 기다리고 timeout을 적용한다. ack가 없거나 불일치하면 맹목
  재시도하지 말고 중단·보고한다. tmux/TUI 어댑터에서는 TUI idle 확인 →
  주입 → terminal tail의 태스크 텍스트 확인으로 수행하며, 파일 기반·다른 CLI
  어댑터에서는 동등한 상태·수신 기록을 사용한다. 주입은 셸 초기화·자동
  업데이트와 레이스해서 조용히 유실될 수 있다(3회 실측) `[OBS]`.
- 작업 명세에 반드시: 목표, 대상 경로(워크트리), 산출 형식, 금지사항
  (push·외부 API·스코프 밖 파일), 완료 보고 형식.
- 격리: 병렬 파일 변경은 워커별 워크트리. 베이스 브랜치는 명시(기본 브랜치
  포인터가 릴리스 브랜치를 가리키는 레포가 실존한다) `[AHE-B3] [OBS]`.
- 장수 워커 vs 일회성 워커: 맥락 축적이 이득인 반복 도메인 작업은 장수
워커에, 단발 작업은 깨끗한 컨텍스트의 일회성 워커에 `[AHE-B3] [AHE-B5]`.

## 5. 근거 (리서치 인용)

방법 주기: 수용한 19개 출처의 URL, 접속 확인일, 주장별 근거는 동반 리서치
문서에 정리했다. 여기서는 그 문서로 연결되는 간결한 교차 참조만 둔다. 별도의
검증 투표를 새로 했다는 뜻은 아니다. `[OBS]`는 이 운영 환경에서 직접 측정한
사실에만 사용한다.

- `[AHE-A1]` context rot: 컨텍스트가 길어질수록 회상이 약해지며, 장기 작업에는
  compaction과 구조화된 메모가 필요하다.
- `[AHE-A2]` 영속 메모리: 활성 컨텍스트 밖에 지식을 저장하고 필요할 때만 불러온다.
- `[AHE-A3]` 레포지토리 지식을 SSOT로 삼으려면 짧은 지도, 구조화된 문서, 기계적인
  최신성 검사가 필요하다.
- `[AHE-A4] [AHE-A5] [AHE-C2]` 체크포인트, condenser, append-only 이벤트 이력은
  지속화와 복구의 경계를 구체화한다.
- `[AHE-B2]` 폭주 사례: 초기 리서치 시스템이 간단한 질문에 서브에이전트 50개를
  띄웠고, 이후 effort 단계와 출처 품질 기준을 넣었다.
- `[AHE-B2]` 모호한 위임은 중복 작업과 빈틈을 만들었다. 목표·산출 형식·도구·출처
  지침·경계를 작업 명세에 넣어야 한다.
- `[AHE-B1]` 평가 루프에는 명시적 기준이 필요하고, 에이전트는 ground truth와
  최대 반복 정지 조건을 사용해야 한다.
- `[AHE-B2]` 인용된 시스템의 멀티에이전트 리서치는 일반 대화 대비 약 15배 토큰을
  사용했다. 실제 비용은 작업과 모델에 따라 달라진다.
- `[AHE-B3] [AHE-B5]` 깨끗한 컨텍스트, 제한된 권한, 영속 정체성, 일회성 워커는
  공개 구현에서 확인되는 운영 패턴이다.
- `[AHE-C1] [AHE-C2] [AHE-C3] [AHE-C4] [AHE-C5]` lifecycle hooks, 이벤트 이력,
  interrupt, 시간 제한, 하트비트, patrol 역할은 루프를 제어하는 수단이다.
- `[OBS]` 런타임 생존 확인, 프롬프트 주입 레이스, diff 범위 리뷰, 크레딧 소진,
  결정 메모리 누락은 하루 운영에서 직접 확인한 사고다.

## 6. 하루 운영에서 나온 함정 목록 (실측)

- 런타임 태스크 조회의 거짓 사망 신호 → 살아 있는 모니터를 죽인 사고. 생존
  확인은 프로세스 실측으로 `[OBS]`.
- 프롬프트 주입과 셸 초기화의 레이스 → 태스크 유실 3회. 수신 검증 의무화 `[OBS]`.
- 배포 설정 리뷰를 diff 범위로 한정 → 이미지 생산 경로(compose) 누락을 리뷰
